## Introduction

Most teams adding safety and cost controls to their AI applications reach for a proxy-based solution. The pitch sounds reasonable: route every request through a sidecar that inspects payloads, enforces policies, and logs results. In practice, proxies introduce latency, become a single point of failure, and still can't see what happens inside your process.

Captar takes a different approach. Instead of sitting between your app and the model, Captar runs guardrails **in-process**, right alongside your application code. No network hop, no deserialization overhead, no blind spots.

## The Problem with Proxy-Based Guardrails

A proxy sits on the network path between your application and the LLM provider. That architectural decision comes with real trade-offs:

1. **Latency** – Every request traverses an extra network hop. For streaming responses, this adds perceptible delay to every token.
2. **Observability gaps** – A proxy sees the request and the response, but it cannot instrument individual tool calls, span-level timing, or intermediate state inside a multi-step agent loop.
3. **Operational complexity** – Proxies require separate deployment, scaling, and monitoring. They add a failure mode that is external to your application.
4. **Policy latency** – Budget checks that run over the network respond too slowly to stop an in-flight generation.

## Budget Guardrails that Actually Enforce Limits

Captar's budget guardrails run inside your process. When you configure a spend limit, Captar checks the cumulative cost of tokens **before** each upstream call. If the budget is exhausted, the call is rejected immediately—no wasted tokens, no surprise bill.

```ts
import { captar } from '@captar/sdk';

const result = await captar.run({
  model: 'gpt-4o',
  budget: { maxCostUsd: 0.05 },
  messages: [{ role: 'user', content: 'Summarize this document' }],
});
// If the budget is exceeded, result.status === "budget_exceeded"
```

Because the check is synchronous and in-process, enforcement is deterministic. There is no window where a request can slip through because the proxy hasn't processed the previous response yet.

## Real-Time Tool Tracking

Agents that call tools need visibility into every invocation: which tool was called, with what arguments, how long it took, and whether it succeeded. A proxy sees the final response but not the individual tool calls inside a multi-step run.

Captar instruments each tool call as a **span** within the trace. You get structured data for every invocation without any manual logging on your part.

```ts
const result = await captar.run({
  model: 'gpt-4o',
  tools: [searchTool, databaseTool],
  messages,
});

for (const span of result.spans) {
  if (span.kind === 'tool') {
    console.log(`${span.name}: ${span.durationMs}ms`);
  }
}
```

This information is available immediately after the run completes, and it flows directly into Captar's trace export pipeline.

## Trace Export

Every run in Captar produces a trace composed of nested spans—model calls, tool invocations, and custom spans you define. Traces are the foundation for debugging, evaluations, and compliance.

Captar can export traces in OpenTelemetry-compatible format, making it straightforward to ship data to your existing observability stack (Jaeger, Grafana Tempo, Honeycomb, or any OTel-compatible backend).

```ts
import { captar } from '@captar/sdk';

captar.configure({
  export: {
    otel: { endpoint: 'http://localhost:4318/v1/traces' },
  },
});
```

No proxy required. The SDK emits traces from inside your process, ensuring zero data loss even under high throughput.

| Operational footprint | Separate service | SDK dependency       |
| Data loss under load  | Possible         | None                 |

If you're building AI-powered features and need cost controls, tool observability, and trace export without adding infrastructure, Captar's local-first approach gives you all three with none of the proxy overhead.
