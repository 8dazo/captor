## The Reliability Gap in AI Apps

Shipping an AI feature is the easy part. Keeping it reliable as prompts change, models update, and edge cases accumulate—that's where most teams struggle. Traditional software has tests, CI, and monitoring. AI applications need the same discipline, but adapted for non-deterministic outputs.

Captar closes this gap with a three-stage pipeline: **traces → datasets → evaluations**. Each stage builds on the previous one, turning raw telemetry into structured quality assurance.

## Stage 1: Span-First Traces

A trace in Captar is not a flat log entry. It's a tree of **spans**, each representing a meaningful unit of work—a model call, a tool invocation, or a custom operation you define.

```ts
import { captar } from '@captar/sdk';

const result = await captar.run({
  model: 'gpt-4o',
  tools: [searchTool, calculatorTool],
  messages: [{ role: 'user', content: "What's 2% interest on $50,000?" }],
});
```

This single run might produce a trace like:

```
run (root span)
  ├── model-call "gpt-4o" (45ms, 320 tokens, $0.003)
  ├── tool-call "calculator" (12ms, result: 1000)
  └── model-call "gpt-4o" (38ms, 180 tokens, $0.002)
```

Every span captures timing, token usage, cost, input/output payloads, and status. This structure is rich enough to answer any debugging question: which step was slow, which tool failed, how much did the full run cost.

## Stage 2: Curated Datasets

Traces are valuable on their own for debugging, but their power multiplies when you curate them into **datasets**. A dataset is a named collection of trace records that represent scenarios you care about—golden paths, known failure modes, edge cases, or regression basins.

Creating a dataset is straightforward:

1. Run your application with Captar tracing enabled.
2. In the Captar dashboard, select the traces that represent important scenarios.
3. Tag them into a named dataset, e.g., `math-qa-golden` or `tool-failure-edge-cases`.

Datasets serve as the ground truth for evaluations. They capture not just the input and output, but the full execution context—every tool call, every intermediate model response, every timing measurement.

## Stage 3: Manual Evaluations

With datasets in place, you can run **manual evaluations** to check whether changes to your prompts, model configuration, or tool definitions produce acceptable results.

A manual eval in Captar works like this:

1. Select a dataset.
2. Optionally configure a new model, prompt, or set of tools to compare against the recorded baseline.
3. Review the outputs side by side, flagging any regressions or improvements.

```ts
const eval = await captar.evaluations.create({
  dataset: 'math-qa-golden',
  runner: {
    model: 'gpt-4o',
    tools: [searchTool, calculatorTool],
    prompt: 'You are a helpful math tutor.',
  },
});
```

Because each entry in the dataset already contains the full trace, you can compare not just final outputs but the entire execution path. Did the agent take a different sequence of tool calls? Did it spend more tokens on a step that used to be simple? These are the questions that matter for reliability, and Captar makes them answerable.

## The Feedback Loop

The pipeline is inherently circular:

```
traces → datasets → evals → new traces → updated datasets → …
```

Every evaluation run produces new traces. Those traces feed back into datasets, which seed the next round of evaluations. Over time, your dataset grows to cover more scenarios, and your confidence in the system's reliability grows with it.

This is the same discipline that makes traditional software test suites effective, but adapted for AI's non-deterministic nature. Instead of binary pass/fail assertions, you get structured comparisons grounded in real execution data.

## Getting Started

1. **Add the SDK** – `npm install @captar/sdk` and instrument your application with `captar.run()`.
2. **Collect traces** – Every run automatically produces a span-first trace.
3. **Curate datasets** – Pick representative traces in the dashboard and tag them into datasets.
4. **Run manual evals** – Compare outputs across model versions, prompts, or tool configurations.

From debugging a single slow run to systematically evaluating prompt changes, Captar gives you a path from traces to confidence.
